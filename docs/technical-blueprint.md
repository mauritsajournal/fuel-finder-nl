## 1. Architecture Overview

The system follows a **static site + data pipeline + edge services** architecture. The frontend is a static SvelteKit site on GitHub Pages. Data is fetched and processed by GitHub Actions (cron), stored in Cloudflare R2 (free object storage), and served to the browser as tiled JSON files. A Cloudflare Worker proxies routing API requests to protect the API key.

### Key Architectural Decisions

1. **No traditional backend server.** GitHub Actions for data processing, Cloudflare R2 for storage, Cloudflare Worker for API proxying -- all free tier.
2. **Data stored in Cloudflare R2, not git.** Avoids repo bloat from committing multi-MB JSON files every 4 hours. R2 free tier: 10GB storage, 10M reads/month.
3. **Viewport-based data loading.** Data is split into geographic tiles. The browser loads only tiles visible in the current map viewport, keeping initial payload under 500KB.
4. **Routing proxied through Cloudflare Worker.** Protects the OpenRouteService API key from client-side exposure and abuse. Worker adds the key server-side.
5. **Fuel data is pre-fetched, not real-time.** Updated every 4 hours via cron. Acceptable for PoC -- fuel prices change 1-2x/day at most stations.
6. **Map tiles from OpenFreeMap.** Zero cost, no API key, no rate limits. Vector tiles for dark theme styling. Fallback: OSM raster tiles.

## 3. Component Design

### 3.1 Data Pipeline (GitHub Actions)

- **Responsibility:** Fetch fuel prices, EV charger data, and POI data from external APIs; process, validate, split into geographic tiles, and upload to Cloudflare R2.
- **Inputs:** API responses from ANWB, OpenChargeMap, Overpass API
- **Outputs:** Tiled JSON files uploaded to Cloudflare R2: `tiles/fuel/{grid_x}_{grid_y}.json`, `tiles/ev/{grid_x}_{grid_y}.json`, `tiles/poi/{grid_x}_{grid_y}.json`, `metadata.json`
- **Dependencies:** Python 3.12+, requests library, boto3 (S3-compatible for R2), GitHub Actions
- **Schedule:** Every 4 hours for fuel + EV data; daily for POI data (changes rarely)
- **Error handling:** Validate response schema before processing. If an API fails, retry 3x with backoff, then keep previous data in R2. Never overwrite with empty/corrupt data. Log all errors.

**Data processing steps:**
1. Fetch ANWB fuel stations (geobox query covering all NL, split into ~6 regions)
2. Normalize price formats (EUR per liter, 3 decimal places)
3. Validate data schema (station must have id, lat, lng, at least one price)
4. Fetch OpenChargeMap data for NL (bounding box query, max 10000)
5. Fetch Overpass API for POIs -- scoped to relevant types only (toilets near highways, rest areas, limited playgrounds near fuel stations)
6. Split all data into geographic grid tiles (0.5 degree grid cells)
7. Generate metadata.json with last_updated timestamps, station counts, tile index
8. Upload changed tiles to Cloudflare R2 via S3-compatible API
9. Update metadata.json in R2

### 3.2 Map Component

- **Responsibility:** Render the interactive map with all markers (fuel stations, EV chargers, POIs). Handle user interaction (pan, zoom, tap marker).
- **Inputs:** JSON data files, user location (Geolocation API), filter state
- **Outputs:** Rendered map with clustered markers, popups on tap
- **Dependencies:** MapLibre GL JS, OpenFreeMap tiles
- **Implementation details:**
  - Use MapLibre GL JS `GeoJSONSource` for data layers
  - Cluster fuel station markers at low zoom levels
  - Different marker icons/colors per category (fuel, EV, POI types)
  - Color-code fuel stations by price (green = cheap, red = expensive)
  - Dark map style to match glassmorphism theme
  - Fly-to animation when selecting a station

### 3.3 Station Detail Panel

- **Responsibility:** Show detailed information about a selected station in a slide-up panel.
- **Inputs:** Selected station data, user location
- **Outputs:** Rendered detail view with prices, facilities, route button
- **Dependencies:** Map component (for selection events), Routing service
- **UI elements:**
  - Station name and brand
  - All fuel prices with relative indicators (cheap/average/expensive)
  - Address and distance from user
  - Opening hours (if available)
  - Facilities (toilets, shop, car wash, etc.)
  - "Navigate" button (triggers route calculation)
  - "Open in Google Maps/Waze" external link

### 3.4 Routing Service

- **Responsibility:** Calculate driving routes from user location to selected station. Display route on map with time and distance.
- **Inputs:** User coordinates (from GPS), station coordinates
- **Outputs:** Route polyline on map, driving time, distance
- **Dependencies:** Cloudflare Worker (routing proxy) -> OpenRouteService Directions API
- **API contract:** GET `https://routing.fuel-finder.workers.dev/route?start={lng},{lat}&end={lng},{lat}` (Worker adds ORS API key server-side)
- **Error handling:** Cache results in sessionStorage (key = start+end coords, 1hr TTL). Show "route unavailable" on API error. Fallback to straight-line distance with estimated time.
- **Rate limiting:** Debounce route requests (500ms). Cache aggressively. Worker enforces 10 req/min per IP.

### 3.5 Filter & Search Component

- **Responsibility:** Allow users to filter map markers by fuel type, price range, distance, and category (fuel/EV/POI). Search by location name.
- **Inputs:** User interactions, current filter state
- **Outputs:** Updated filter state (triggers map re-render)
- **Implementation:**
  - Fuel type toggle: Euro 95 | Diesel | LPG | Electric
  - Sort/filter: Cheapest | Nearest | Best value
  - Distance radius slider: 5km - 50km
  - Location search: Geocoding via OpenRouteService geocode API (free)

### 3.6 Data Store

- **Responsibility:** Load tiled JSON data from Cloudflare R2, provide reactive data access to all components, handle viewport-based tile loading.
- **Inputs:** Tiled JSON files fetched from R2 based on map viewport
- **Outputs:** Reactive Svelte stores for all data entities
- **Implementation:**
  - Svelte 5 `$state` runes for reactive data
  - Fetch metadata.json on app load (~1KB) to get tile index and timestamps
  - Load data tiles on demand based on current map viewport (MapLibre `moveend` event)
  - Cache loaded tiles in memory (Map<tileKey, data>)
  - Pre-process on load: calculate distances from user, sort by price
  - GeoJSON conversion for MapLibre consumption
  - Show "last updated X hours ago" from metadata timestamps

## 4. Data Architecture

### 4.1 Data Model

```
FuelStation {
  id: string                    // Unique identifier (source_prefix + original_id)
  name: string                  // Station name/brand

  }[]
  pricing?: {

  }

}

  id: string
  type: "playground" | "toilet" | "rest_area" | "picnic_site" | "viewpoint"

}

  lastFuelUpdate: string        // ISO 8601
  lastEVUpdate: string

  chargerCount: number
  poiCount: number

```

### 4.2 Storage Strategy

All data is stored as tiled JSON files in Cloudflare R2:

```
R2 bucket: fuel-finder-data/
  metadata.json                 // Tile index, timestamps, counts (<5KB)
  tiles/
    fuel/
      51.0_3.5.json             // Fuel stations in grid cell (lat_lng)
      51.0_4.0.json
      51.5_3.5.json
      ...                       // ~20-30 tiles covering NL
    ev/
      51.0_3.5.json             // EV chargers in same grid
      ...
    poi/
      51.0_3.5.json             // POIs in same grid
      ...
```

**Tiling strategy:**
- Grid cells of 0.5 degrees latitude x 0.5 degrees longitude
- Netherlands covers roughly 50.7-53.6N, 3.3-7.3E = ~24 grid cells
- Each fuel tile: ~100-200 stations, ~20-50KB
- Each EV tile: ~500-2000 chargers, ~50-200KB
- Each POI tile: varies, ~10-50KB
- Initial viewport load: 2-4 tiles = ~100-400KB (acceptable on 4G)

**Size optimization:**
- Truncate coordinate precision to 5 decimal places (~1m accuracy)
- Keep human-readable property names (gzip handles compression well)
- R2 serves with appropriate Cache-Control headers (max-age=3600)

### 4.3 Data Flow

```
[API Sources] ---(GH Actions, every 4h)---> [Tiled JSON in Cloudflare R2]
[Browser] ---(fetch metadata.json on load)---> [Tile index, timestamps]
[Browser] ---(fetch tiles on viewport change)---> [Visible data tiles from R2]
[Browser] ---(Svelte stores)---> [Map Component, Detail Panel, Filters]
[User taps station] ---(CF Worker proxy)---> [OpenRouteService API] ---> [Route on map]
```

## 5. Integration Design

### 5.1 ANWB Fuel Prices API

- **System:** ANWB Onderweg backend
- **Protocol:** REST (GET)
- **Authentication:** None (public API)
- **Endpoint:** `https://api.anwb.nl/routing/points-of-interest/v3/all?type-filter=FUEL_STATION&bounding-box-filter={south},{west},{north},{east}`
- **Data exchanged:**
  - Request: Bounding box coordinates covering NL
  - Response: JSON array of fuel stations with id, name, address, coordinates, prices array
- **Error handling:** Retry 3x with exponential backoff. If all fail, keep previous data.
- **Rate limits:** Unknown (not documented). Use conservative approach: single request per grid cell, 1 second delay between requests.
- **Notes:** NL needs to be split into ~4-6 bounding boxes due to geobox size limit. Merge results and deduplicate.

### 5.2 DirectLease TankService API

- **System:** DirectLease TankService backend (tankservice.app-it-up.com)
- **Protocol:** REST (GET)
- **Authentication:** X-Checksum header (SHA-1 based, algorithm documented in pyfuelprices)
- **Endpoints:**
  - Station list: `https://tankservice.app-it-up.com/Tankservice/v2/places?fmt=web&country=NL&country=BE&lang=en`
  - Station detail: `https://tankservice.app-it-up.com/Tankservice/v2/places/{station_id}?_v48&lang=en`
- **Data exchanged:**
  - Request: GET with checksum header
  - Response: JSON with station data; prices as integers (divide by 1000)
- **Error handling:** If checksum fails, log error and skip (ANWB is primary source)
- **Rate limits:** Unknown. Fetch station list once, then detail pages in batches with delay.

### 5.3 OpenChargeMap API

- **System:** OpenChargeMap global registry
- **Protocol:** REST (GET)
- **Authentication:** API key (free registration)
- **Endpoint:** `https://api.openchargemap.io/v3/poi/?output=json&countrycode=NL&maxresults=10000&compact=true&verbose=false&key={API_KEY}`
- **Data exchanged:**
  - Request: Country filter, max results
  - Response: JSON array of charging locations with coordinates, connectors, operator, status, usage cost
- **Error handling:** Retry 2x. OpenChargeMap is generally reliable.
- **Rate limits:** API key required for >250 results. No documented rate limit for free tier but be respectful (max 1 request per fetch cycle).

### 5.4 OpenStreetMap Overpass API

- **System:** Overpass API (public OSM query interface)
- **Protocol:** REST (POST with Overpass QL query)
- **Authentication:** None
- **Endpoint:** `https://overpass-api.de/api/interpreter`
- **Data exchanged:**
  - Request: Overpass QL query for amenity=toilets, leisure=playground, tourism=picnic_site, highway=rest_area in NL bounding box
  - Response: JSON with nodes and their tags
- **Error handling:** Overpass can be slow under load. Set 60s timeout. Retry once.
- **Rate limits:** Shared public resource. Max 2 queries per fetch cycle. Run POI fetch daily (data changes rarely).
- **Query notes:** Scope POIs to be relevant for drivers. Fetch toilets and rest areas broadly (useful along highways). Limit playgrounds to those near fuel stations or highways (a playground in a residential neighborhood is noise). Viewpoints and picnic sites only near major roads.
- **Query example (toilets + rest areas):**
  ```
  [out:json][timeout:60];
  area["ISO3166-1"="NL"]->.nl;
  (
    node["amenity"="toilets"](area.nl);
    node["highway"="rest_area"](area.nl);
    node["highway"="services"](area.nl);
    node["tourism"="picnic_site"]["highway"](area.nl);
  );
  out center;
  ```
- **Playground query:** Separate query, filtered to within 500m of a fuel station or highway exit. Keeps dataset manageable.

### 5.5 OpenRouteService Directions API (via Cloudflare Worker Proxy)

- **System:** Cloudflare Worker -> OpenRouteService hosted API
- **Protocol:** REST (GET)
- **Authentication:** ORS API key added by Worker (never exposed to client)
- **Client endpoint:** `https://routing.fuel-finder.workers.dev/route?start={lng},{lat}&end={lng},{lat}`
- **Worker logic:**
  1. Validate start/end coordinates are within NL bounding box
  2. Rate limit: 10 requests/minute per IP (using CF Worker KV or in-memory)
  3. Add ORS API key from Worker environment variable
  4. Forward to: `https://api.openrouteservice.org/v2/directions/driving-car?api_key={KEY}&start={lng},{lat}&end={lng},{lat}`
  5. Return ORS response to client
- **Data exchanged:**
  - Request: Start and end coordinates
  - Response: GeoJSON with route geometry, duration (seconds), distance (meters)
- **Error handling:** Worker returns appropriate error codes (400 for invalid coords, 429 for rate limit, 502 for ORS failure). Client shows "route unavailable" toast and falls back to straight-line distance.
- **Rate limits:** ORS: 2,000/day, 40/min. Worker adds per-IP limiting (10/min). Client debounces (500ms) and caches in sessionStorage.

### 5.6 Cloudflare R2 Data Storage

- **System:** Cloudflare R2 object storage
- **Protocol:** S3-compatible API (for uploads), HTTPS (for reads)
- **Authentication:** S3 access key/secret for GitHub Actions uploads; public read via R2 public URL
- **Endpoints:**
  - Upload (pipeline): `https://{account_id}.r2.cloudflarestorage.com/fuel-finder-data/`
  - Read (browser): `https://pub-{hash}.r2.dev/` or custom domain
- **Data exchanged:**
  - Upload: Tiled JSON files from GitHub Actions
  - Read: Browser fetches individual tile files based on viewport
- **Error handling:** R2 is highly reliable. Client retries failed tile fetches once. Shows cached data if available.
- **Rate limits:** Free tier: 10GB storage, 10M class B (read) operations/month, 1M class A (write) operations/month. More than sufficient for PoC.

### 5.7 OpenRouteService Geocode API (via Cloudflare Worker Proxy)

- **System:** Cloudflare Worker -> OpenRouteService Pelias geocoding
- **Protocol:** REST (GET)
- **Authentication:** API key added by Worker
- **Client endpoint:** `https://routing.fuel-finder.workers.dev/geocode?text={query}`
- **Worker logic:** Sanitize query text, add API key, add `boundary.country=NL`, forward to ORS
- **Data exchanged:**
  - Request: Search text
  - Response: GeoJSON FeatureCollection with coordinates and place names
- **Error handling:** Show "location not found" in search. Debounce input (300ms).
- **Rate limits:** Shared with directions quota at ORS level. Worker applies same per-IP limiting.

## 6. Security Design

### Authentication & Authorization
- No user accounts, no login. Public read-only application.
- API keys (OpenRouteService, OpenChargeMap) stored as GitHub repository secrets for Actions pipeline.
- Cloudflare R2 credentials stored as GitHub Secrets for data upload.
- No API keys exposed in client-side JavaScript.

### Data Encryption
- All API calls over HTTPS.
- GitHub Pages serves over HTTPS by default.
- Cloudflare R2 serves over HTTPS.
- No sensitive data at rest (public fuel price data only).

### GDPR Compliance
- No cookies set by the application.
- No analytics/tracking scripts.
- Geolocation permission requested only on user action ("Find me" button).
- Location data stays in the browser. Routing requests go through our Cloudflare Worker (no logging of coordinates).
- Third-party tile requests expose user IP to OpenFreeMap. Document in a minimal privacy notice.

### Secret Management
- ORS API key: stored in Cloudflare Worker environment variable (never in client JS)
- OpenChargeMap API key: stored in GitHub Secrets (server-side pipeline only)
- R2 access credentials: stored in GitHub Secrets (pipeline upload only)
- Cloudflare account token: stored in GitHub Secrets

### Input Validation
- Cloudflare Worker validates coordinates are within NL bounding box before forwarding to ORS
- Sanitize search input before sending to geocode API
- Data pipeline validates API response schemas before uploading to R2
- Frontend validates tile JSON structure on load (reject corrupt tiles, show error)
- Rate limiting: Worker enforces 10 routing requests/minute per IP

### Legal Risk Mitigation
- ANWB and DirectLease APIs are undocumented/reverse-engineered
- PoC is acceptable for technical demonstration
- Production use would require formal data licensing agreements
- Data source is modular: can swap providers without frontend changes
- Document data provenance in metadata.json

```yaml

jobs:

    runs-on: ubuntu-latest

      R2_ACCESS_KEY_ID: ${{ secrets.R2_ACCESS_KEY_ID }}
      R2_SECRET_ACCESS_KEY: ${{ secrets.R2_SECRET_ACCESS_KEY }}
```
